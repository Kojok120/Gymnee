## 1. プロダクト概要

カレンダーを起点に、筋トレに関する記録・共有・購買が一つのアプリで完結する iOS アプリ。

**価値提案 / 差別化**
- **クロスジム継続性**：ジムを変えても、合トレで一時的に別ジムへ行っても、記録は同一アプリ・同一タイムライン上に残る。ジムは「アカウントの属性」ではなく「来店（visit）の属性」として扱う。
- **写真チェックイン**：ジムで写真を撮ることがチェックインそのもの。来店記録が残り、同時に SNS 共有導線が立ち上がる。
- **カレンダー起点**：ホームはカレンダー。記録の閲覧・計画・継続可視化の全てがカレンダー上で完結する。
- **L3 ログ**：セット / レップ / 重量レベルのフルワークアウト記録を最初から備える。

**ターゲット**：ジムに通う習慣を持ち、トレーニング内容を記録・継続・可視化したい個人。複数ジム利用者・出張/合トレが多い層と特に相性が良い。

## 2. スコープ

**v1 に含む（フルスコープ）**

1. L3 ワークアウトログ（前回値オートフィル / ルーティン / レストタイマー / PR / 1RM / ボリューム / プレート計算）
2. カレンダー × 継続（ヒートマップ / 予定→実績 / 連続記録 / リカバリービュー / ジム別サマリー）
3. 写真・チェックイン・ソーシャル（GPS 自動検出 / ジム図鑑 / 進捗写真 / 合トレ / 共有カード）
4. データ分析（ダッシュボード / バランス可視化 / CSV エクスポート）
5. iOS ネイティブ機能（HealthKit / ウィジェット / Watch / Live Activity / Siri）
6. コマース（プロテイン等の購買 / 補給ロギング→在庫リマインド / ゴール連動レコメンド / サブスク）

**v1 で対象外（明示）**
- Android 版（将来検討）
- コミュニティ大規模機能（DM、コメントスレッド、ハッシュタグ検索等）
- 栄養素フル計算（PFC 自動計算は v1 では行わず、補給ロギングは「摂取した/しない」と量の記録に留める）

## 4. データモデル

主要エンティティと中心カラム。すべて `user_id` を持ち RLS で本人のみアクセス可（共有対象は除く）。

### 4.1 ユーザー・ジム

| エンティティ | 主要カラム | 備考 |
|---|---|---|
| `profiles` | id, display_name, avatar_url, bio, created_at | Supabase auth user に 1:1 |
| `gyms` | id, name, chain, address, lat, lng, source(`preset`/`user`), created_by | プリセット＋ユーザー作成の両方。来店から参照 |
| `gym_equipment` | id, gym_id, label, note | ジム別の設備メモ（任意・将来コミュニティ化余地） |

### 4.2 来店・ワークアウト（コア）

| エンティティ | 主要カラム | 備考 |
|---|---|---|
| `visits` | id, user_id, gym_id, visited_at, photo_url, lat, lng, note | **写真チェックイン**の本体。1 来店 = 1 レコード |
| `visit_partners` | id, visit_id, partner_user_id | 合トレタグ付け。双方の来店として表示 |
| `workouts` | id, user_id, visit_id(nullable), date, name, routine_id(nullable), note | セッション。来店に紐付くが、ジム外記録も許容 |
| `exercises` | id, name, muscle_group, equipment, is_custom, created_by(nullable) | 種目マスタ。プリセット＋ユーザー作成 |
| `workout_exercises` | id, workout_id, exercise_id, order_index | セッション内の種目並び |
| `exercise_sets` | id, workout_exercise_id, set_index, weight, reps, rpe(nullable), type(`normal`/`warmup`/`drop`/`superset`), is_pr | L3 の最小単位 |
| `routines` | id, user_id, name | ルーティン/テンプレ |
| `routine_exercises` | id, routine_id, exercise_id, order_index, target_sets | テンプレの構成 |
| `personal_records` | id, user_id, exercise_id, type(`max_weight`/`max_reps`/`est_1rm`/`max_volume`), value, achieved_at, workout_id | PR。検出時に記録／更新 |

### 4.3 身体・写真・分析

| エンティティ | 主要カラム | 備考 |
|---|---|---|
| `body_metrics` | id, user_id, date, weight, body_fat, measurements(jsonb) | 体重・体脂肪・各部位サイズ。HealthKit と双方向 |
| `progress_photos` | id, user_id, date, photo_url, visibility(`private`/`friends`/`public`) | 体型写真。既定 `private` |

### 4.4 ソーシャル

| エンティティ | 主要カラム | 備考 |
|---|---|---|
| `follows` | id, follower_id, followee_id, created_at | フォロー関係 |
| `feed_items` | id, user_id, type(`visit`/`pr`/`workout`), ref_id, created_at, visibility | フィード生成元 |

### 4.5 コマース

| エンティティ | 主要カラム | 備考 |
|---|---|---|
| `products` | id, name, description, price, image_url, category, goal_tags(text[]), stock(nullable) | 在庫管理の有無はフルフィルメント方式に依存 |
| `orders` | id, user_id, status, total, stripe_payment_intent, created_at | |
| `order_items` | id, order_id, product_id, quantity, unit_price | |
| `supply_logs` | id, user_id, product_id(nullable), date, amount | 補給ロギング。在庫リマインドの根拠 |
| `subscriptions` | id, user_id, tier, status, started_at | 有料プラン（採用可否は **⚠️要決定**） |

> リレーション要点：`visits 1—* workouts`、`workouts 1—* workout_exercises 1—* exercise_sets`、`gyms 1—* visits`、`exercises 1—* workout_exercises`。ジムは来店経由でしか紐付かないため、ジムを跨いでも `visits` の連続性が保たれる（＝差別化の中核）。

## 6. 機能要件（詳細）

### 6.1 認証・オンボーディング
- Sign in with Apple を主、メールを副とする。
- 初回に位置・カメラ・通知・HealthKit の利用目的を提示し許諾を取得（拒否時もコア機能は動作）。

### 6.2 カレンダーホーム
- 月/週切替。来店日にマーカー、ワークアウト有無を区別表示。
- **ヒートマップ**：来店頻度を貢献グラフ風に可視化（年間ビュー）。
- **予定 → 実績**：未来日にワークアウトを計画し、当日に消化チェック。
- 連続記録・週次頻度ゴール（例：週3）と達成率。

### 6.3 チェックイン
- カメラ起動 → 撮影 → ジム選択。選択時、**GPS ジオフェンスで現在地のジムを自動補完**（候補提示）。
- 任意メモ、合トレ相手のタグ付け。
- 保存後、共有カード生成導線を提示（§6.6）。

### 6.4 ジム管理 / 図鑑
- ジムマスタはプリセット（主要チェーン）＋ユーザー自己登録の併用。**プリセットの初期収録範囲は ⚠️要決定**。
- ジム別サマリー：単一タイムライン上で、ジムごとの来店回数・最頻ジムを集計。
- **ジム図鑑**：訪問済みジムをコレクションとして可視化（出張・合トレと好相性）。

### 6.5 ワークアウトログ（L3・最重要）
- **前回値オートフィル**：種目選択時に前回のセット（重量・レップ）を初期表示。L3 継続の生命線。
- ルーティン/テンプレからワンタップ開始、または空で開始。
- **レストタイマー**：セット完了で起動、通知付き。Live Activity 連動（§6.10）。
- セット種別：normal / warmup / drop / superset、RPE・RIR 任意入力。
- **PR 自動検出**：max_weight / max_reps / est_1rm / max_volume を更新時に検出し記録、通知・共有カード連動。
- 推定 1RM（Epley 等）、週あたり部位別ボリューム集計。
- プレート計算機：目標重量に対する片側プレート構成を提示。

### 6.6 共有カード・SNS 導線
- 写真上にサマリ（種目 / 連続日数 / PR / ジム名）をオーバーレイした共有カードを生成。テーマ・表示項目を選択可能。
- ネイティブ共有シート経由で共有（Instagram Stories 含む）。**各 SNS の直接投稿仕様は流動的なため、共有シート方式を基本とし、特定 SNS の専用 API には依存しない**。

### 6.7 進捗写真・身体メトリクス
- 体型写真は来店写真と別枠、既定 `private`、月次比較タイムライン。
- 体重・体脂肪・各部位サイズを記録し推移表示。HealthKit と同期（§6.9）。

### 6.8 分析ダッシュボード
- 強度進捗（種目別重量推移）、頻度、部位バランス（レーダーチャート、脚の偏り等）。
- **リカバリービュー**：直近で鍛えた部位を可視化し、回復済み＝次やる候補を提示（ボリュームと連動）。
- **CSV エクスポート**：全記録をエクスポート可能（データ所有権）。

### 6.9 HealthKit 連携
- 読込：体重・体組成。書き戻し：ワークアウト（種別・時間）、必要に応じアクティブエネルギー。
- 許諾は最小権限・用途明示。

### 6.10 iOS プラットフォーム機能
- **ウィジェット**：連続日数・最終ワークアウト・次の予定（ホーム/ロック画面）。
- **Live Activity**：レストタイマー／アクティブセッションの動的表示。
- **Apple Watch**：手首からのクイックチェックイン、セッション中心拍記録。
- **Siri ショートカット**：「ジムに着いた」で即チェックイン（App Intents）。
- **ジオフェンス自動チェックイン**：登録ジム接近時にチェックインを提案。

### 6.11 ソーシャル
- フォロー/フォロワー、フィード（来店・PR・ワークアウトを可視性設定に従い表示）。
- 合トレ相手のタグ付け（`visit_partners`）で双方に記録が残る。
- **フィードの公開範囲（公開 / 友達限定）は ⚠️要決定**。

### 6.12 コマース
- 商品一覧・詳細・カート・**Stripe 決済**・注文履歴。物理グッズのため Apple IAP は対象外（外部決済）。
- **補給ロギング → 在庫リマインド**：摂取/購入サイクルから「そろそろ無くなりそう」を提示し購入導線へ。
- ゴール連動レコメンド（増量フェーズなら高カロリー寄り等、`goal_tags` ベース）。
- サブスク（高度分析・無制限写真・上位カード等）。**デジタル課金は IAP 要件に該当しうるため、サブスク採用可否と決済経路は ⚠️要決定**。
- **フルフィルメント方式（自社在庫 / ドロップシップ等）が未定。在庫管理・配送の有無を規定するため ⚠️要決定**。

## 7. 非機能要件

- **オフラインファースト**：ネット未接続でも記録可能。ローカル（SwiftData）を正とし、復帰時に Supabase へ同期。コンフリクトは last-write-wins を基本とし、セット記録は追記型で衝突を避ける。
- **パフォーマンス**：カレンダースクロールとセット入力のレイテンシを体感ゼロに近づける。入力は楽観的更新。
- **プライバシー / 法令**：体型写真は既定非公開。位置・カメラ・Health は用途明示＆オプトイン。個人情報保護法（APPI）準拠、データ削除・エクスポートを提供。
- **セキュリティ**：Supabase RLS で行レベル権限。画像はアクセス制御付き URL。Sign in with Apple。
- **アクセシビリティ**：Dynamic Type、VoiceOver、十分なコントラスト。
- **ローカライズ**：日本語を主、英語対応可能な構造（文言外部化）。
- **App Store 準拠**：位置/カメラ/写真/Health の用途記述（Info.plist）、物理グッズの外部決済、Sign in with Apple 提供。
- **テスト**：ドメインロジック（PR 検出・1RM・ボリューム集計・同期）にユニットテスト。

## 8. 実装順序（依存順フェーズ）

スコープを削るのではなく、**依存関係に沿った実装順**を定義する。各フェーズは前段の成果物に依存する。

| Phase | 内容 | 依存理由 |
|---|---|---|
| **P0 基盤** | Xcode プロジェクト構成、Supabase 接続、認証（Sign in with Apple）、データモデル＋ SwiftData ローカルストア＋同期エンジン、ナビゲーション骨格 | 以降すべての前提。データモデルと同期がここで固まる |
| **P1 コアループ** | カレンダーホーム、チェックイン（カメラ＋ジム選択＋GPS）、来店記録、ジムマスタ（プリセット＋自己登録） | プロダクトの背骨。P0 の認証・モデルに依存 |
| **P2 L3 ログ** | 種目マスタ、ワークアウトセッション、前回値オートフィル、レストタイマー、ルーティン、PR 検出、1RM/ボリューム、プレート計算 | `workouts`/`exercise_sets` モデル（P0）と来店連携（P1）に依存 |
| **P3 共有・写真** | 共有カード生成、ネイティブ共有シート、進捗写真 | 来店写真（P1）とワークアウトサマリ（P2）が揃ってから |
| **P4 データ・ヘルス** | HealthKit 連携、身体メトリクス、分析ダッシュボード、CSV エクスポート、ヒートマップ、リカバリービュー | 分析は P2 の記録蓄積が前提（データが無いと無意味） |
| **P5 iOS 機能** | ウィジェット、Live Activity、Apple Watch、Siri、ジオフェンス自動チェックイン | コアループ（P1/P2）の上に乗る拡張 |
| **P6 ソーシャル** | フォロー、フィード、合トレタグ、ジム図鑑 | 来店・PR・ワークアウト（P1〜P3）を素材にする。比較的独立 |
| **P7 コマース** | 商品カタログ、カート、Stripe 決済、注文、補給ロギング→在庫リマインド、ゴール連動レコメンド、（任意）サブスク | 最も独立かつ重い。決済・フルフィルメント決定後に着手 |

> 依存の要点：**P0 → P1 → P2 が直線的な背骨**。P3 以降は背骨の上に概ね並列で積める。コマース（P7）とソーシャル（P6）は他機能への依存が最も小さい一方で外部要因（決済・公開範囲）の決定待ちが発生しやすいため、最後段に置く。

## 9. 未確定事項 / 要決定リスト

着手前に解消すべき判断。

1. **最低 iOS バージョン**：SwiftData 採用なら iOS 17 以上。リーチとのトレードオフ。
2. **画像ストレージ**：Supabase Storage か Cloudflare R2 か。
3. **ジムプリセットの初期収録**：主要チェーンをどこまで初期データとして持つか、それとも自己登録のみで開始するか。
4. **コマースのフルフィルメント方式**：自社在庫 / ドロップシップ等。在庫管理・配送フローの有無を規定。
5. **サブスク採用可否と決済経路**：デジタル課金は Apple IAP 要件に該当しうるため、採用するか、する場合の決済経路を確定（物理グッズの Stripe とは別判断）。
6. **ソーシャルの公開範囲**：フィードを公開とするか友達限定とするか。
7. **同期コンフリクト方針**：last-write-wins を基本とする想定だが、確定が必要。
